# /apply-review

GitHubのPull Requestに付けられたレビューコメントを読み込み、指摘の妥当性を評価した上で、適切なものへの修正コード生成と、不適切なものへの返信ドラフト作成を行う。

## 使い方

```
/apply-review <PR番号またはURL>
```

## 手順

以下のステップを順番に実行してください。

### Step 1: PRの情報を取得する

引数として渡された `$ARGUMENTS` をPR番号またはURLとして扱い、`gh` コマンドでPRの詳細とレビューコメントをすべて取得してください。

```bash
# PR情報とレビューコメント取得
gh pr view $ARGUMENTS --json title,body,headRefName,baseRefName,files
gh pr reviews $ARGUMENTS --json body,state,author
gh pr comments $ARGUMENTS --json body,author,path,line
```

取得できたPRの概要（タイトル・ブランチ・変更ファイル一覧）を表示してください。

### Step 2: レビューコメントを分類する

取得したすべてのレビューコメントに対して、以下の基準で妥当性を評価し分類してください。

**対応すべき指摘（修正対象）:**
- バグ・ロジックエラーの指摘
- セキュリティ上の問題
- パフォーマンスの改善提案
- コーディング規約・スタイルの統一
- 可読性・保守性の改善
- テストの追加・修正

**返信で説明すべき指摘（議論対象）:**
- 設計方針の違いによる意見の相違
- 既存の制約・仕様によりすでに意図的に選択した実装
- 誤解に基づいたコメント
- スコープ外の改善提案（別PRでの対応が適切）

分類結果を一覧表示し、各コメントに「✅ 修正対象」「💬 議論対象」のラベルを付けて示してください。

### Step 3: 対象ファイルのコードを読み込む

修正対象に分類したコメントが指している各ファイルを `Read` ツールで読み込んでください。
コメントに行番号が含まれる場合は、その前後20行程度を確認してください。

### Step 4: 修正を実装する

修正対象コメントそれぞれについて、実際にコードを修正してください。

- `Edit` ツールを使ってファイルを直接修正する
- 複数ファイルにまたがる修正も漏れなく行う
- 修正後は「どのコメントに対応したか」「何を変更したか」を簡潔に説明する

### Step 5: 議論対象コメントへの返信ドラフトを作成する

議論対象に分類したコメントそれぞれについて、以下の形式で返信ドラフトを作成してください。

```
---
[コメント内容の要約]

**返信ドラフト:**
（既存の設計意図・制約・理由を説明し、なぜ現在の実装が適切かを丁寧に述べる。
または、別PRで対応することを提案する。）
---
```

### Step 6: 作業サマリーを表示する

最後に以下の形式でサマリーを表示してください。

```
## 作業サマリー

### 修正したコメント (N件)
- [ファイル名:行番号] コメントの要約 → 変更内容

### 返信ドラフトを作成したコメント (N件)
- コメントの要約 → 返信方針

### 変更ファイル
- ファイルパス (変更の概要)
```

---

**注意事項:**
- レビュアーの意図を正確に理解してから修正に着手すること
- 修正により既存のテストが壊れていないか確認すること
- セキュリティ・バグ修正は優先度高として必ず対応すること
- 返信ドラフトは丁寧かつ建設的なトーンで作成すること
