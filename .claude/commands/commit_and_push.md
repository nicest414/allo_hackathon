以下の手順でコミットとプッシュを行ってください。

## 手順

### 1. 現在の状態を確認

以下を並列で実行してください：
- `git status` — 変更ファイルの確認（`-uall` フラグは使わない）
- `git diff` — 未ステージの差分
- `git diff --staged` — ステージ済みの差分
- `git log --oneline -5` — 直近のコミット履歴（コミットメッセージのスタイル確認用）

### 2. 変更をグループに分類する

差分を分析し、変更の「種類」ごとにグループ分けしてください。

**分類の基準（異なるグループにすべきもの）：**
- 機能追加（add）
- バグ修正（fix）
- リファクタリング（refactor）
- テスト追加・修正（test）
- ドキュメント（docs）
- 設定・依存関係の変更

**分類の判断ポイント：**
- 1つのコミットで revert したとき意味のある単位になっているか
- 変更の理由（why）が1つの文で説明できるか
- 例：機能追加とそのテストは同じグループ。機能Aの修正と無関係な機能Bのリファクタは別グループ

グループが1つなら通常のコミット1本。複数なら分割コミットを提案する。

### 3. ユーザーに確認

グループ分けの結果をユーザーに提示してください：

**グループが1つの場合：**
- コミットメッセージ案とステージ対象ファイルを提示
- 承認を得る

**グループが複数の場合：**
- 「以下の X 件のコミットに分けることを提案します」と明示
- 各コミットについて「対象ファイル」と「コミットメッセージ案」を列挙
- ユーザーが分け方を変更できるようフィードバックを受け付ける

**コミットメッセージの方針：**
- 1〜2文で変更の「なぜ（why）」を中心に説明
- `add` / `fix` / `refactor` / `test` / `docs` など変更の性質を先頭に置く
- 過去のコミット履歴のスタイルに合わせる

**ユーザーが承認するまでコミット・プッシュを実行しないでください。**

### 4. コミットとプッシュ

ユーザーの確認が取れたら、以下を順番に実行してください：

1. グループの順番通りに、1コミットずつ処理する
   - 対象ファイルだけをステージ（`.env` や認証情報ファイルは除外する）
   - コミットを作成

```bash
git add <対象ファイル>
git commit -m "$(cat <<'EOF'
コミットメッセージ

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"
```

2. 全コミット完了後にリモートにプッシュ

```bash
git push -u origin HEAD
```

3. プッシュ完了をユーザーに伝える（作成したコミット数・各ハッシュ・ブランチ名を含める）
