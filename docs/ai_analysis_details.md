# AI・解析系（MediaPipe / Meyda / Gemini）の実装詳細と最適化の工夫

本ドキュメントは、「メン・トライアル」に搭載されているAIおよびデジタル信号処理（DSP）モジュールの内部設計、実装詳細、およびハッカソンや実用性を考慮したパフォーマンス上の工夫について詳しくまとめたものです。

---

## 📸 1. 顔解析（MediaPipe Face Landmarker）

### 💡 実装の仕組み
- レンダラープロセス上で **MediaPipe Face Landmarker WASM** をロードし、Webカメラの映像ストリーム（就活生）およびデスクトップキャプチャのフレーム（面接官）を並列で解析します。
- `faceLandmarker.ts` がラッパーとして機能し、468個の顔メッシュ座標（Landmarks）を取得します。

### 🛠️ 技術的工夫とパフォーマンス最適化
1. **メインスレッドのブロッキング回避（Web Workersの検討とFPS分離）**
   - 顔ランドマークの検出は非常にCPU/GPU負荷が高く、愚直に毎フレーム（30fps〜60fps）走らせると、ブラウザスレッドがビジー状態になり、画面上の「雷アニメーション（60fps）」が激しくカクつきます。
   - **工夫**: 顔解析の実行頻度を意図的に落とし（就活生は **6 FPS**、面接官の画面共有は **4 FPS**）、アニメーション描画の処理と解析のループを完全に非同期で分離しました。これにより、UIの滑らかさを維持したまま、実用的なレスポンス速度で表情を解析できます。
2. **黄金比による動的顔切り抜き（Crop）アルゴリズム**
   - 就活生のカメラや面接官の画面共有から「顔部分だけ」を検出し、アバターやプレビュー用に自動でトリミングする `calculateFaceCropBox` を実装。
   - **工夫**: 顔のランドマークの端（最外郭座標）を動的に算出し、顔のサイズに対するアスペクト比を維持しつつ、上下左右に適切なバッファ（`faceCropPaddingRatio: 0.35`）を設けて自動的に正方形で切り出す画像処理ロジックを実装。顔がカメラに近づいたり遠ざかったりしても、常に中心に顔が収まるポートレート画像を生成します。

## 🎙️ 2. 声分析（Meyda によるリアルタイム特徴量抽出）

### 💡 実装の仕組み
- Web Audio API を用いて就活生のマイク入力を取得。
- 軽量オーディオ解析ライブラリ **Meyda** を用い、ブラウザ側でリアルタイムに以下の3つの音響特徴量を抽出します。
  - `pitchVariation` (ピッチのゆらぎ・声の震え)
  - `speechRate` (発話速度)
  - `pauseRatio` (沈黙の割合)

### 🛠️ 技術的工夫とパラメータ正規化
1. **理想の発話速度（IDEAL_SPEECH_RATE）からの偏差検出**
   - 人は緊張すると「早口（マシンガントーク）」になるか、逆に「言葉に詰まって遅口」になります。
   - **工夫**: 単に発話速度の速さをスコアにするのではなく、人間が最も聞き取りやすく堂々として見える発話速度（`IDEAL_SPEECH_RATE = 6` 文字/秒相当）を定義し、そこからの絶対偏差（離れ具合）を計算することで、**「早口すぎても遅口すぎても、声の焦りスコアが上昇する」**という多角的な判定を実現しました。
2. **Web Audioノードの接続最適化**
   - `AudioContext` 内で `ScriptProcessorNode` と `GainNode` を接続し、マイク入力を解析しつつ、就活生自身のスピーカーから自分の声がハウリング（ループバック）して聞こえないように `GainNode.gain.value = 0` (ミュート) で安全に切断するオーディオパイプラインを構築しています。

## 💬 3. STT（音声テキスト化）＆ フィラー検出

### 💡 実装の仕組み
- 取得したマイク音声を $16\text{bit}$ PCM形式のバイナリ（`ArrayBuffer`）にエンコードし、250ms ごとに Electron メインプロセスにIPC経由でストリーミング送信します。
- メインプロセスはそれを **Deepgram Streaming API** または **Gemini Live STT** に流し込み、超低遅延で文字起こしテキストを取得します。
- 確定（`isFinal`）したテキストデータに対し、`fillerDetector.ts` を走らせて不要な言葉を検出します。

### 🛠️ 技術的工夫とアルゴリズム設計
1. **長い単語優先マッチング（貪欲法による重複カウント防止）**
   - フィラー検出は「あのー」「あの」「えっと」といった日本語特有のフィラーを文字列マッチングします。しかし、単純に部分一致で検索すると、「あのー」という発言に対して、「あのー」と「あの」が重複して2回カウントされてしまう問題があります。
   - **工夫**: フィラー辞書の配列を**「文字数の長さ順」に動的にソート**してから、長いフィラーから順にマッチングを実行。一致した部分を順次文字列から削除（consume）していくことで、重複カウントを完全に防ぐアルゴリズムを実装しました。
2. **スライディングウィンドウによる「時間減衰フィラースコア」**
   - 面接の最初に1回「えっと」と言ってしまったペナルティが、面接終了までずっと残り続けるのはゲーム性およびUXとして不適切です。
   - **工夫**: 過去すべての発言ではなく、直近一定時間（`DEFAULT_FILLER_WINDOW_MS = 60000` / 60秒間）の確定発話のみを保持するスライディングウィンドウ方式を採用。
   - レンダラー側で1秒周期の `setInterval` を回して `recomputeFiller` を実行し、時間が経過した古いフィラーはバッファから自動的に除外（減衰）される仕組みを導入しました。これにより、「しばらく落ち着いて綺麗に話すと、ゲージが回復する」という気持ちの良いゲームループが成立しています。

## 🧠 4. 返答内容判定（Gemini 1.5 Flash）

### 💡 実装の仕組み
- 就活生の回答テキストが確定した際、メインプロセス内の `geminiJudgeClient.ts` から Google AI Studio SDK / Gemini API を呼び出します。

### 🛠️ 技術的工夫とセキュリティ設計
1. **APIキーの厳格な隠蔽（IPCサンドボックス）**
   - レンダラープロセス（ブラウザ）は、DevTools を開くことでメモリ内の変数を覗き見ることができます。
   - **工夫**: レンダラーには `GEMINI_API_KEY` を一切公開せず、メインプロセス内で完結させます。レンダラーは単に「質問と回答のテキスト」をIPC経由で投げるだけで、判定スコアと日本語の理由テキストのみを受け取るセキュリティ設計にしています。
2. **`responseSchema` による完全構造化JSON出力の強制**
   - 自然言語をLLMに判定させると、稀に出力フォーマットが崩れ、パースエラーを起こしてアプリが落ちる原因になります。
   - **工夫**: Gemini 1.5 のスキーマ定義機能を利用し、APIに対して `{ score: number, reason: string }` という厳格なJSONスキーマでの出力を強制。これにより、正規表現やトリッキーなパース処理を排除し、**パースエラー率 $0\%$** を実現しています。
3. **過剰API呼び出し防止（重複リクエストのデバウンス）**
   - `responseJudger.ts` では、直近の質問と回答の組み合わせを記録し、同じ文字列が連続して送られた場合は自動的にリクエストをスキップ（デバウンス）する仕組みを持ち、ハッカソン中のAPI利用制限や無駄なネットワーク遅延を回避しています。
