# プロダクト概要：就活面接 優劣判定オーバーレイ（仮）

本プロダクトは、就活の面接中にリアルタイムで就活生と面接官の「優勢/劣勢」を判定し、**「逆転裁判」風のUIでオーバーレイ表示するデスクトップアプリ**です。

---

## 💡 主要な機能とUX要件

1. **画面オーバーレイ表示**
   - ZoomやGoogle MeetなどのWeb会議画面に重ねて表示する。
   - アプリケーションウィンドウは**常に最前面・背景透過・クリック透過**になっており、操作を邪魔しない。特定の操作パネル（ボタンや入力欄）にホバーした時のみクリック可能になる。
2. **リアルタイム優勢度（0〜100）の判定**
   - 就活生と面接官のデータから、リアルタイムに戦況を数値化する。
3. **逆転裁判風の演出**
   - **`DominanceClashBanner`**: 画面上部で就活生（左）と面接官（右）の雷がせめぎ合い、優勢度に応じて中央の衝突位置が左右に変動する。
   - **カットイン演出**: 優勢度が特定のしきい値を超えて振り切れた際、カットインアニメーションと効果音（「異議あり！」風）を再生する。

---

## 🛠️ 入力ソースと分析項目

| 分析項目 | 取得方法 / ライブラリ | 対象者 | 判定される内容 |
| :--- | :--- | :--- | :--- |
| **顔の表情・笑顔・焦り** | **MediaPipe Face Landmarker** | 就活生 / 面接官 | 表情変化、笑顔の有無、顔の焦り具合（瞬きや眉の動き） |
| **声の焦り** | **Meyda** (Web Audio API) | 就活生 | ピッチ変動、発話速度、沈黙（ポーズ）の割合 |
| **フィラー検出** | **STT（音声認識）** + 文字列マッチ | 就活生 | 「えっとですね」「あのー」などの不要な言葉の検出 |
| **返答内容の良し悪し** | **STT** $\rightarrow$ **Gemini 1.5 Flash** | 就活生 | 面接官の質問に対する就活生の回答の適切さ（論理的整合性など） |

---

## 💻 技術スタック

- **シェル (デスクトップラッパー)**: Electron (OSネイティブの透過・最前面制御のため)
- **音声ループバック取得**: `electron-audio-loopback` (macOSのScreenCaptureKitを使用し、面接官の声をキャプチャ)
- **描画・フロントエンド**: Vite + React + TypeScript
- **アニメーション・SE**:
  - `Rive` (ベクターアニメーション / カットイン用)
  - `framer-motion` (UIトランジション)
  - `howler` (効果音・BGM再生)
- **AI・機械学習関連**:
  - `Gemini 1.5 Flash` (構造化出力 `responseSchema` を用いた返答内容の評価)
  - `Deepgram streaming` または `Gemini Live STT` (リアルタイム音声テキスト化)
- **状態管理**: `Zustand` (レンダラープロセス内の軽量状態管理)

## 📂 ディレクトリ構成と責務

本プロダクトは、Electronの**メインプロセス（Main）**、**プリロード（Preload）**、**レンダラープロセス（Renderer）**で明確に責務を分離しています。

```
src/
├── main/                           # ===== メインプロセス：OS API / 秘密鍵の管理 =====
│   ├── index.ts                    # エントリーポイント（ウィンドウ生成・ライフサイクル管理）
│   ├── env.ts                      # 環境変数・APIキーの読み込み（レンダラーには渡さない）
│   ├── windows/                    # オーバーレイウィンドウの設定（透過・最前面）
│   ├── audio/                      # 面接官側の音声キャプチャ (electron-audio-loopback)
│   ├── stt/                        # STTプロバイダ (Deepgram / Gemini) の抽象化
│   └── llm/                        # Gemini Flashによる返答内容の評価クライアント
│
├── preload/                        # ===== プリロード：レンダラーへ安全なAPIを露出 =====
│
├── renderer/                       # ===== レンダラープロセス：Vite + React UI =====
│   └── src/
│       ├── capture/                # カメラ・マイク・画面キャプチャの生データ取得
│       ├── analysis/               # MediaPipe / Meyda による特徴量抽出
│       ├── services/               # メインプロセスへのIPC通信窓口 (薄いプロキシ)
│       ├── domain/scoring/         # ★純粋なロジック層 (顔・声・STT・Gemini結果のスコア化)
│       ├── store/                  # Zustandによる優勢度状態管理
│       └── ui/                     # Reactコンポーネント (せめぎ合う雷バナー、手動判定パネル)
│
└── shared/types/                   # ===== 共通型定義 (IPCチャネル名、データスキーマ等) =====
```

## ⚖️ 優勢度計算の重み付け (DOMINANCE_WEIGHTS)

`src/renderer/src/domain/scoring/dominanceCalculator.ts` にて以下のように設定されています。

- **就活生の表情 (顔スコア)**: $30\%$
- **面接官の表情 (顔スコア)**: $15\%$
- **就活生の声の焦り度**: $20\%$ (焦りが少ないほど優勢)
- **フィラーの多さ**: $15\%$ (フィラーが少ないほど優勢)
- **返答内容の評価 (Gemini)**: $20\%$

## ⚠️ 未確定・未実装の検討事項 (READMEより)

1. **STTプロバイダの決定**: `Deepgram streaming` vs `Gemini Live STT` (リアルタイム性とフィラーの残存度で比較検証が必要)
2. **特徴量 $\rightarrow$ スコアの変換ロジック**: MediaPipeのランドマーク座標から「笑顔・焦り」への変換、およびMeydaのピッチ・速度から「声の焦り」への変換ロジックの設計。
3. **優勢度計算の調整**: 各特徴量をどうスコア化し、合成するかの重み付けのチューニング。
4. **カットイン発生のしきい値**: 「優勢度が振り切れた」と判定する条件の定義。
5. **Windows対応**: ScreenCaptureKitはmacOS専用のため、Windows対応時のループバック音声キャプチャ手法の検討。
6. **Gemini判定のタイミング**: 発話の終わりを検知して投げるか、バッチ処理で定期的に評価するかの設計。
